Opening Other Apps

  • Ti.Platform.openURL()
  • Need to know the "URL scheme"
  • On Android, launch an intent directly

Do not include the slashes with geo: URLs on Android but you must provide something for the lat/long (0,0). Also on Android, you can also use http:// with a maps.google.com URL to open the Maps app, though users might be prompted to choose between Maps and the browser. On some devices, geo: might open Google Earth.

Example

Open the native maps app

function doClickMaps(e) {  
  if(OS_IOS) {
    Ti.Platform.openURL('maps://maps.google.com/maps?q=440+Bernardo+Ave,+Mountain+View,+CA');
  } else {
    // no slashes, 0,0 could be a lat/long or use address as shown
    Ti.Platform.openURL('geo:0,0?q=440+Bernardo+Ave,+Mountain+View,+CA');
    // on Android, can also use http:// ...
    Ti.Platform.openURL('http://maps.google.com/maps?q=440+Bernardo+Ave,+Mountain+View,+CA');
  }
}
For Android, if you provide an address, enter the lat/long as 0,0 like shown. Otherwise, leave off the ?q= and address and just provide a lat/long pair.

Driving Directions (navigation)


// iOS - Apple maps
http://maps.apple.com/?daddr=San+Francisco,+CA&saddr=cupertino

// iOS - Google maps
comgooglemaps://?daddr=440+Bernardo+Ave,+Mountain+View,+CA&directionsmode=driving


// Android - Google maps
http://maps.google.com/maps?daddr=440+Bernardo+Ave,+Mountain+View,+CA&directionsmode=driving
// optional, add saddr for source 'address', can use lat/long instead of address
On iOS, you can launch driving directions in either Apple Maps or Google Maps. Either accept street addresses, though according to Apple's docs, not latitude/longitude pairs.

On Android, you should use lat/long for the destination and source points. You can use street addresses, however, that will launch a browser session from which accessing the Navigation app requires many clicks.

Topic

Custom URL Schemes

Custom URL Schemes

  • Let other apps launch yours
  • Need to select your own 'myappscheme://' URL scheme string
  • No registration or oversight, so choose carefully to avoid conflicts
  • Requires you to configure the tiapp.xml with custom platform-specific application properties
You can define your own custom URL schemes so that other apps can launch yours. Typically, you would do so in order to enable one of your apps to launch another. But you can share the URL scheme to let 3rd party developers launch your app as well.

Custom iOS URL Schemes


<ios>
  <plist><dict>
  <key>CFBundleURLTypes</key>
  <array>
    <dict>
      <key>CFBundleURLName</key>
        <string>com.company.yourapp</string>
      <key>CFBundleURLSchemes</key>
        <array>
          <string>Urlschemes</string>
        </array>
  </dict>
  </array>
  </dict></plist>
</ios>
  • App ID must match your app's App ID
  • 'Urlschemes' is the custom URL Scheme, should be lead-capped
Add this to your tiapp.xml or create a custom Info.plist. The CFBundleURLName should match your app's App ID. The Urlscheme should be lead-capped and something that's likely to be unique to your app.

Custom Android URL Schemes

  • Modify AndroidManifest.xml
  • Either via tiapp.xml or /platform/android/AndroidManifest.xml
  • Modify the activity node that you want launched — it doesn't have to be your main activity
  • Lots of extra options — see the Google docs

Manifest Example


<activity android:name=".UrlschemesActivity"
  android:label="URLSchemes" android:theme="@style/Theme.Titanium"
  android:configChanges="keyboardHidden|orientation">
  <intent-filter>
    <action android:name="android.intent.action.MAIN" />
    <category android:name="android.intent.category.LAUNCHER" />
    </intent-filter>
  <intent-filter>
    <action android:name="android.intent.action.VIEW"/>
    <category android:name="android.intent.category.DEFAULT"/>
    <category android:name="android.intent.category.BROWSABLE"/>
    <data android:scheme="urlschemes" android:host=""/>
  </intent-filter>
</activity>
  • You're enabling a special intent filter
  • Use action type VIEW to launch your app
  • 'urlschemes' is your URL scheme, by convention it's lowercase
Key here are the tags in the second intent-filter block. Of them, the data tag defines the URL scheme name. The other tags set options that control how this activity will be launched and whether the user will have access to the rest of the app once this activity is launched. See the Google docs for more information on these options.

Checking if Target App Exists

  • iOS: Use Ti.Platform.canOpenURL()
  • Android: Ti.Platform.openURL() returns a Boolean; check for false to know if app was opened
  • Then, prompt user to download the necessary app
  • Could open the app store for them!

Example


function doClick(e) {  
  if(OS_IOS) {
    if(Ti.Platform.canOpenURL('Urlschemes://')) {
      Ti.Platform.openURL('Urlschemes://');
    } else {
      alert('You must install the Target app first');
    }
  } else {
    var didItWork = Ti.Platform.openURL('urlschemes://');
    if(!didItWork) {
      alert('You need to install the Target app');
    }
  }
}

Topic

App Arguments

Arguments

  • Pass data to another application when you launch it via its URL scheme
  • Work like query parameters on a 'normal' URL
  • You get the entire URL in the target app & need to parse out the arguments
  • Of course, how you do this varies by platform

Launch Arguments on iOS

  • Use Ti.App.getArguments() in the target application
  • Use a 'resumed' app event listener to get arguments when app launched from backgrounded state

function grabArguments() {
  var args = Ti.App.getArguments();
  // data validation, parsing
}
grabArguments();
Ti.App.addEventListener(
   'resumed', grabArguments);
Take note that we're listening for the resumed not resume event. App arguments are not available with the resume event.

Launch Arguments on Android

  • Access via the current Activity
  • Need a handle to the Intent that launched the activity
  • Data available even if app launched from backgrounded state

var activity = Ti.Android.currentActivity;
activity.addEventListener("create", function(e) {
  var args = activity.getIntent().getData();
  // validate, parse, handle
});

Parsing Arguments


function urlParser(url) {
  url = url.replace(/[Uu]rlschemes:\/\/\?/,"");
  return url.split('&');
}
Here's a sample function for parsing out the arguments from the URL scheme itself. The function returns an array of arguments. Replace [Uu]rlschemes with your custom scheme.

"Sanitize" your inputs

Just like with web applications, you should:

  • Check for correct data types
  • Don't 'blindly' pass arguments to database queries
  • Confirm permissions and security
  • Don't expose destructive actions via launch arguments
We have to take care to sanitize any arguments received by our apps to be sure that we don't accidentally expose destructive actions. Apps are subject to the same types of SQL injection and parameter manipulation attacks that web applications must endure.
